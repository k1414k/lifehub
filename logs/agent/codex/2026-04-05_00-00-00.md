## ビルドエラーの原因

### 主因: `postcss.config.mjs` (ESM) が Turbopack に認識されない

Next.js 15 では `next dev` が **Turbopack** をデフォルトで使用する。
Turbopack は PostCSS の設定ファイルを **CommonJS (`postcss.config.js`)** から読み込む仕様であり、
ESM 形式の `postcss.config.mjs` を認識しない。

その結果、PostCSS が適用されず `globals.css` の `@tailwind base;` ディレクティブが
Turbopack / webpack のパーサーに素の CSS として渡り、以下のエラーが発生していた:

```
Module parse failed: Unexpected character '@' (1:0)
./src/app/globals.css
@tailwind base;
```

`next build`（webpack）はローカル環境では `postcss-load-config v6` が `.mjs` を動的 import
で処理できるため偶然通過していたが、Docker 上の `next dev`（Turbopack）では確実に失敗する構成だった。

### 副因: `next-auth` の未使用残存 (EvalError の潜在リスク)

`next-auth` は `package.json` に `dependencies` として記載されていたが、
ソースコードのどこにも import がなかった。

`next-auth` v4 は内部で `eval()` / `new Function()` を使用するコードを含むため、
何らかのきっかけで middleware のバンドルに混入すると
`EvalError: Code generation from strings disallowed for this context`
（Edge Runtime が eval を禁止するため）が発生する原因になりうる。

削除することで将来的なリスクも解消した。

## 実装理由・判断根拠

- `postcss.config.js` (CJS) は webpack / Turbopack / postcss-load-config の全バージョンで動作する最も安全な形式
- `.mjs` に変更されていた経緯は不明だが、Next.js 15 への移行時に Turbopack との非互換が生じたと推定
- `next-auth` は AGENTS.md の「要求範囲外の依存は触らない」方針に基づき、未使用ならば削除が適切と判断
