- 変更内容の要約:
  - 会員登録失敗の直接原因を backend のログから特定し、登録後に Devise が session へ書き込もうとして `ActionDispatch::Request::Session::DisabledSessionError` で 500 になっていた問題を修正した。
  - 登録時の `sign_up` を `sign_in(..., store: false)` に差し替え、API モードでも JWT を発行できるようにした。
  - Devise の `skip_session_storage` を `http_auth` のみから `http_auth` / `params_auth` へ広げ、API リクエストで session に依存しない設定へ寄せた。
  - `sign_out` は frontend から必ず `DELETE /api/v1/auth/sign_out` を呼ぶように変更し、backend でも Devise の destroy フローを通しつつ `request.session_options[:skip] = true` で session 書き込みを回避した。
  - `respond_to_on_destroy` のシグネチャ不一致で 500 になっていた問題も追加で修正し、runtime でも JWT revocation が成立する状態にした。
  - auth response に `created_at` を追加し、`User` 型と auth response shape の不整合を解消した。
  - frontend の token 管理は `localStorage` 直保存 + Zustand persist の二重管理をやめ、Zustand persist (`sessionStorage`) の token 1 系統へ整理した。
  - 認証済み判定用の軽量 cookie (`lifehub_auth=1`) を frontend 側で同期し、Next middleware が protected route を未認証時に `/auth/login?next=...` へリダイレクトできるようにした。
  - `AuthProvider` を追加し、hydration 後に token がある場合のみ `/api/v1/me` で current user を復元する初期化フローを実装した。
  - API 401 は interceptor 側で auth state を破棄し、auth page (`/auth/login`, `/auth/register`) と通常 protected API の挙動が競合しないよう `skipAuthRedirect` を導入した。
  - ログイン / 登録フォームは backend のエラーメッセージを表示できるよう改善し、成功時は `next` クエリを尊重して遷移するようにした。
  - Header に正式なログアウト UI を追加し、API 失敗時でも client state が破綻しないよう finally で state を破棄する構成にした。

- 実装理由または判断根拠:
  - まず最優先の会員登録失敗は、frontend の payload や strong params 不一致ではなく、ユーザー作成後の Devise 標準 `sign_up` が API モードで session を触り 500 を起こしていることが development log の先頭例外で確認できた。
  - localStorage に JWT を直置きし続けるのは避けたかったため、今回は token の source of truth を `sessionStorage` に寄せ、`/me` 復元で user を再構築する構成へ変更した。
  - HttpOnly Cookie ベースへ JWT 全面移行するには backend 側で cookie からの JWT 抽出、CSRF 方針、devise-jwt の受け口整理まで必要になり今回の修正範囲を超えるため、今回は「JWT の localStorage 依存排除」「状態二重化解消」「middleware 保護」「正式 sign_out」を優先した。
  - middleware の cookie は機密情報ではなく route guard 用の sentinel とし、秘密そのものは保持しないようにした。

- 未対応事項や注意点:
  - JWT 自体はまだ frontend が保持して Authorization header に載せる構成のため、完全な HttpOnly Cookie 認証には移行していない。
  - token の保存先は `localStorage` から `sessionStorage` へ改善したが、XSS 耐性の観点では「JS から読める token をゼロにした」わけではない。
  - middleware の cookie は route guard 用の補助情報であり、server 側で JWT 真偽を検証する本物の認証境界ではない。API 保護の本体は引き続き backend の JWT 認証。
  - `docker compose exec backend bundle exec rspec ...` 実行時に Rails binstub の警告は出るが、spec 自体は成功している。今回の修正対象外のため未対応。
