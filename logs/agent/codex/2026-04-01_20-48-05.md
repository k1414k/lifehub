- 理由 / 判断根拠:
  - 収入 / 支出履歴は「保有資産の評価額推移」と意味が異なるため、既存 `transactions` を流用せず専用モデルへ分離した
  - 総資産はユーザー入力ではなく、各資産項目の最新値から自動導出する方が仕様に合い、整合性も崩れにくいため保存しなかった
  - グラフは平均化せずイベントベースの階段線にするため、期間開始時点の補完と未記録期間 `null` 扱いをフロントの集計ロジックに集約した
  - 既存の収支データは新仕様へ安全に意味変換できないため、旧 `transactions` は破壊せず残し、新 UI / 新 API を資産側へ切り替える移行方針を採用した
  - UI はモバイルでも崩れないよう、カード・フォーム・チャートを縦積み前提で組み、広い幅では 2 カラム以上に展開する構成にした
